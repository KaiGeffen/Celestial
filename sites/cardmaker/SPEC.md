# Card Maker — Spec

A web app where anyone can compose a custom Celestial card — name, stats,
theme, and rules text — then publish it to a public gallery, download it, or
share a link. Fan cards spread on Discord and socials; every one of them is an
ad with our card frame on it.

Status: **spec / not yet built**

## Goals

1. Anyone (no account, no game knowledge) can make a plausible-looking
   Celestial card in under a minute.
2. The result is pixel-faithful to real cards: same frame layers, same fonts,
   same stat placement, same gold keyword highlighting, at the game's native
   card resolution.
3. Cards can be **published to a shared gallery** where other visitors browse
   recent creations. A published card has a permalink.
4. Output is also locally shareable: PNG download and copy-to-clipboard.

## Non-goals

- No user-uploaded images — a card is entirely defined by its fields
  (name, cost, points, theme, text). This keeps storage tiny and the
  moderation surface text-only.
- No accounts (publishing is anonymous; an optional free-text "creator" field
  at most).
- No importing custom cards into the game.
- No card-balance validation (any numbers are allowed — it's a toy).
- No comments, likes, or other social features on the gallery (v1).

## Card anatomy (must match the game)

The game composes a card from layers (see `client/src/lib/cardImage.ts`), and
the card maker reproduces the same stack on a `<canvas>`:

| Layer (bottom → top) | Source                         | Notes                               |
| -------------------- | ------------------------------ | ----------------------------------- |
| Background           | `card/background/{theme}.webp` | Theme = 0–8                         |
| Subject (art)        | `subjects/{name}.webp`         | Picked from a curated set; see below|
| Arc                  | `card/arc/{theme}.webp`        |                                     |
| Container            | `card/container/{theme}.webp`  | Frame chrome; sits over the art     |
| Cost                 | text overlay                   | Top-left, game's stat font + stroke |
| Points               | text overlay                   | Below cost, same treatment          |
| Name                 | text overlay                   | Times New Roman (system font)       |
| Rules text           | text overlay                   | Wrapped in the text box; see below  |

- **Subject picker:** the user picks the card's art from a curated set of the
  game's existing subject arts (the game ships 123; the curated list can be
  any size up to all of them). Subject arts are full 472 × 672 layers,
  pre-positioned to sit under the frame — they stack with no clipping or
  placement math. The curated list lives in `gameData.json` and is
  **append-only**, so a published card's `subject` index always resolves to
  the same art. Default subject: `Dove` (the game's own fallback for unknown
  cards).
- Native size: **472 × 672** (the game's 2× card assets). The canvas renders at
  this size; the on-page preview may display smaller, but export is always
  472 × 672.
- Text styles copy the game's (`client/src/settings/style.ts` `cardTitle`,
  and the stat/text styles used by `CardImage`), including stroke colors.
- Card text renders in **Times New Roman** (system font stack `'Times New Roman', Times, serif` — no font files shipped).
- Layer assets are **copied into `sites/cardmaker/assets/`** — per
  the sites rule, never referenced from the game's `/assets/` path. When card
  frames change in the game, the copies here are refreshed manually (add a
  line item to the game's release checklist).

### Rules text: gold keywords and card references

Matching `CardImage.createText` exactly:

- **Keywords** (`Nourish`, `Birth`, …, optionally followed by a number, e.g.
  `Nourish 3`) render in the game's gold — **`#FABD5D`** — using the same
  word-boundary regex semantics as the game
  (`\b{keyword}(?:[ ]*(-?\d+))?(?=\b|[.,!?;])`).
- **References to real card names** appearing in the rules text render in the
  same gold (`\b{cardName}\b`).
- All rules text carries the game's black stroke.
- The keyword list and card-name list are exported from `shared/` into a
  small JSON in `sites/cardmaker/assets/` (generated by a script or copied
  manually alongside the frame assets). The site must not import from
  `shared/` at runtime — it's a static page.
- Canvas implication: the text renderer must support inline color runs
  (tokenize → measure → draw per run, wrapping at the text-box width). No
  hover hints on the gold text (that's a game feature, out of scope here).

## UI / behavior

Single page, two panes (stacked on mobile), plus a gallery section below.

**Left — the form**

- **Name** — text, max ~24 chars (auto-shrink to fit the nameplate like the
  game does, then hard-cap).
- **Cost** — integer 0–9.
- **Points** — integer 0–9.
- **Rules text** — textarea, max ~200 chars, wrapped into the text box, with
  gold keyword/reference highlighting per above.
- **Theme** — 9 swatches (themes 0–8), showing each frame's palette.
- **Subject** — thumbnail picker over the curated subject arts (searchable by
  name if the set is large). Thumbnails lazy-load so the curated set's size
  doesn't slow first paint.
- **Creator** (optional) — free text, max ~20 chars, shown in the gallery.
- **Reset** — returns to the default card.

**Right — the card**

- Live canvas preview updating on every input.
- **Download** — exports the canvas as `{card-name}.png` at 472 × 672.
- **Copy** — copies the PNG to the clipboard (feature-detected; hidden where
  unsupported).
- **Publish** — saves the card's fields to the gallery (see API below) and
  shows the permalink (`/cardmaker/?id={id}`) with a copy button.
- **Keyword reminders** — every keyword in the rules text shows its reminder
  line (from `shared/state/keyword.ts`) below the card, X substituted with
  the written value, on a dark backing like the game's hint popups. Like the
  game's `Catalog.getReferencedKeywords`, keywords in the referenced card's
  text are included too.
- **Referenced card** — if the rules text names a real card, that card is
  rendered below (first match), mirroring the game's hint behavior.

**Gallery**

- Below the maker: a grid of recently published cards, newest first,
  paginated ("Load more"). Cards are **rendered client-side from their
  fields** — the list endpoint returns everything needed, no image hosting.
- Clicking a gallery card opens its permalink view: the card large, plus a
  **Remix** button that loads its fields into the maker.
- Opening `/cardmaker/?id={id}` deep-links to that card.

**Search page** (`/cardmaker/search/`)

- A second page for finding cards, with a toggle between two sources:
  - **Game cards** — all collectible cards, searched client-side over
    `gameData.json` using the deck editor's search syntax (`name:`, `text:`,
    `cost:2`, `cost:3+`, `points:1-3`, quoted phrases, `!` negation — ported
    from `client/src/scene/deckEditor/cardSearchFilter.ts`, minus the
    deck-specific `present`). Free text also searches referenced keywords'
    reminder text and referenced cards' text, like the game.
  - **Community cards** — published custom cards, searched via the API
    (`q` param). Clicking one opens it in the maker (`../?id={id}`).
- Results render on half-resolution canvases via the shared renderer.

**Page dressing**

- Header: game logo (from `../about/assets/`), title "Card Maker".
- Footer: "Made with Celestial's card frame — play at celestialdecks.gg",
  Discord link.

## Backend & database

Publishing requires persistence. The frontend stays a static page; a small
API handles publish/browse. Because cards are fields-only, rows are a few
hundred **bytes**, not kilobytes — storage is a non-issue.

- **Database:** the game's existing Postgres, in a new table owned by this
  feature (no FK into game tables):

  ```
  custom_cards
    id          serial primary key
    name        text        -- length-capped
    cost        int         -- 0–9
    points      int         -- 0–9
    text        text        -- length-capped
    theme       int         -- 0–8
    subject     int         -- index into the append-only curated subject list
    creator     text        -- optional, length-capped
    created_at  timestamptz default now()
    hidden      bool default false   -- moderation kill switch
  ```

- **API** (JSON; no auth):
  - `POST /cardmaker/api/cards` — body: the fields. Server re-validates every
    cap (name/text/creator length, cost/points range, theme range, subject
    range). Returns `{ id }`.
  - `GET /cardmaker/api/cards?before={id}&limit=20&q={query}` — newest-first
    page of full card fields (enough to render each card client-side).
    Optional `q` filters by name/text/creator substring (for the search page).
  - `GET /cardmaker/api/cards/{id}` — one card's fields.

- **Where the API lives:** the game server (it already has Express, Drizzle,
  and the deploy pipeline) — new schema entry in `server/src/db/schema` and a
  small router, path-routed in NPM: `/cardmaker/api → backend`. A separate
  microservice is possible later but is not worth a second deploy target now.

- **Abuse controls (required for v1):**
  - Rate limit publishes per IP (e.g. 10/hour) at the API.
  - Length/range caps re-validated server-side.
  - `hidden` flag + a one-line admin SQL/script to hide a card by id; hidden
    cards 404 and drop out of the gallery. The moderation surface is
    text-only (name / rules text / creator), so this plus a "Report a card"
    Discord link in the footer is the entire v1 story.

## Tech

- Frontend: plain HTML + CSS + vanilla JS (one page, no framework, no build
  step) — consistent with the other `sites/`. Rendering via 2D `<canvas>`;
  card text uses the system Times New Roman stack, so no font loading step.
- The *maker* works without the API (download/copy still function if the
  backend is down); publish and gallery degrade with a friendly message.

## Directory & deployment

Unlike the four existing sites (which share one nginx image), the card maker
ships as **its own image**, so it can be deployed/iterated without touching
the marketing sites:

```
sites/cardmaker/
├── SPEC.md            # this file — excluded from the image
├── Dockerfile         # nginx:alpine, COPY . , own nginx.conf
├── nginx.conf         # same caching/404 policy as sites/nginx.conf
├── index.html
├── cardmaker.css
├── cardmaker.js
└── assets/
    ├── card/          # background/, arc/, container/ (themes 0–8), copied from the game
    ├── subjects/      # the curated subject arts, copied from the game
    └── gameData.json  # keyword names, card names, curated subject list (append-only)
```

- Served at **`/cardmaker/`** on the main domain: compose service `cardmaker`
  (image tag `cardmaker`, on `net`, no published ports) + NPM custom
  locations:
  - `/cardmaker     → http://cardmaker:80`
  - `/cardmaker/api → backend` (the game server)
- Deploy order rule from `sites/README.md` applies: container up → NPM routes
  verified → done.
- Add `/cardmaker/` to the sitemap (served from `client/`).
- nginx 404s `SPEC.md`, `Dockerfile`, `nginx.conf` (same policy as the shared
  sites config).

## Acceptance criteria

1. Opening `/cardmaker/` cold shows a complete default card in < 2s.
2. Every form change is reflected on the canvas immediately; keywords and
   real card names in the rules text render gold (`#FABD5D`) exactly as the
   game would render them.
3. Downloaded PNG is 472 × 672 and matches the preview.
4. Publish → permalink → opening the permalink in a fresh browser re-renders
   the identical card from its fields; Remix loads its fields into the maker.
5. Gallery shows newest cards first and paginates.
6. All server-side caps hold when bypassing the UI (curl a 10k-char text /
   theme 99 / subject 9999 → 4xx).
7. With the API down, the maker still fully works for download/copy.
8. Renders correctly on mobile Safari/Chrome (single-column layout).
9. A side-by-side of a real in-game card and the same card built here is
   indistinguishable at a glance (frame, fonts, stat positions, gold text).

## Decided

- **Fields only** — no user image uploads; a card is name + cost + points +
  theme + subject + text (+ optional creator). The gallery stores and serves
  fields; clients render.
- **Subject picker** — card art is chosen from a curated, append-only set of
  the game's existing subject arts. Subjects are full 472 × 672 layers, so
  they stack under the frame with zero placement math.
- **No watermark** on exported or published images.
- Served at **`/cardmaker/`** (no subdomain).
- Gold highlight color is **`#FABD5D`**, hardcoded to match `cardImage.ts`.

## Open questions

1. **Which subjects to curate** — all 123 game arts, or a hand-picked subset?
   All-of-them is simplest and most expressive; the costs are image weight in
   the site (mitigated by lazy-loading thumbnails) and that every art teases
   game content. Kai's call at build time; the append-only list makes it safe
   to start small and grow.
2. **Gallery retention** — rows are tiny, so likely keep everything; revisit
   only if spam volume forces it.
